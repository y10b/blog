# 운영자가 실제로 만들고 있는 Muse 프로젝트(이미지 생성 광고 플랫폼)의 학습/실험 노트입니다.
# AI가 글을 생성할 때 이 노트의 일화·수치·결정 근거를 인용해 1인칭으로 작성합니다.
# 형식: [주제/모델/실험명] 내용

[Muse 프로젝트 개요] Muse는 "업종별로 잘 터지는 광고를 데이터로 설계해주는 플랫폼"이다. 소상공인이 업종을 입력하면 검증된 광고 영상이 만들어지는 구조. 일반인 모델은 얼굴/목소리를 제공하고 로열티를 받고, 소상공인은 검증된 광고를 얻고, 플랫폼은 성과 데이터가 쌓일수록 추천이 정확해진다. 핵심 엔진은 (1) 얼굴 인상 분석 MLP와 (2) 한국인 얼굴 특화 FLUX LoRA 두 가지.

[비즈니스 피벗] 초기에는 "AI 모델을 싸게 만들어주는 플랫폼"이었지만, 팀 논의 후 "업종별로 잘 터지는 광고를 데이터로 설계해주는 플랫폼"으로 피벗했다. 이유: 소상공인의 진짜 고민은 "싸게 만들기"가 아니라 "뭘 만들어야 팔리는지 모르겠다"는 것. 기술이 아니라 결정의 문제였다.

[ImpressionMLP v1] 첫 번째 모델. 입력 65차원(랜드마크 좌표 56d + 인상 수치 9d), 출력 16개 인상 속성(이진 분류), MLP [128, 64] 2층. 결과 Val F1 0.38. 거의 동전 던지기 수준. Validation Loss가 진동하고 과적합 조짐이 보였다.

[F1 0.38 원인 분석 6가지] (1) 정보 부족 — 인상 수치가 9개뿐인데 16개 속성을 판단해야 했다. (2) 판별 불가 속성 포함 — "Attractive(매력)", "Young(젊음)" 같은 건 점 좌표만으로 알 수 없는데 라벨에 들어가 있었다. (3) 클래스 불균형 — "Double_Chin"은 데이터의 5%뿐이라 모델이 무조건 "없다"로 예측해도 정확도 95%. (4) 추론 시 스케일링 미적용 — 학습 때 정규화한 걸 추론 때 안 해서 결과가 엉뚱. (5) 네트워크 너무 얕음 — 2층으로는 복잡한 패턴 학습 불가. (6) 고정 threshold 0.5 — 속성마다 최적 threshold가 다른데 다 0.5로 잘랐다.

[ImpressionMLP v2] 6가지 원인을 다 해결한 버전. 인상 수치 9개 → 45개로 확장(대칭, 각도, 면적, 비율 등). 예측 속성 16개 → 12개(판별 불가 4개 제거). 네트워크 [128, 64] → [256, 128, 64] 듀얼헤드. Loss에 class weight 추가. AdamW + Cosine Annealing. 랜드마크/특징 둘 다 스케일링. 속성별 최적 threshold 탐색. 결과 F1 0.38 → 0.583. +52% 개선.

[듀얼헤드 MLP 설계 결정] 입력을 두 갈래로 나눠 각각 처리한 후 합치는 구조. 랜드마크 좌표(56d) → 별도 임베딩 112d, 기하 특징(45d) → 별도 임베딩 64d, 합쳐서 176d → [512, 256, 128] → 12 출력. 이유: 좌표값과 계산된 비율은 데이터 성격이 다르다. 한꺼번에 넣지 말고 각각의 의미에 맞는 처리를 먼저 한 후 합치는 게 효과적이다.

[45개 기하 특징] 얼굴 점 28개 사이의 관계를 6가지 방식으로 측정. 기본 비율 9개(눈/코/입 크기), 대칭 6개(좌우 차이), 교차 비율 8개(눈 대비 입 크기), 각도 6개(눈썹 아치, 눈꼬리 기울기), 면적 8개(눈/입/코 삼각형 면적), 추가 비율 8개(이마/인중/턱 비율).

[제거한 속성 4개와 이유] (1) Attractive — 주관적, 점 좌표로 불가능. (2) Young — 피부 텍스처/머리색 문제, 기하 특징 아님. (3) Rosy_Cheeks — 순수 색상, 랜드마크와 무관. (4) Bags_Under_Eyes — 미세 텍스처, 점 좌표로 감지 불가. 라벨 정의를 모델 입력 능력과 일치시키는 게 먼저였다.

[ImpressionMLP v3] 비즈니스 피벗 후 라벨 자체를 바꾼 버전. 물리 속성(코가 크다 같은) 12개 이진 분류 → 인상 점수(친근함, 신뢰감 같은) 10개 회귀. Loss BCELoss → MSELoss. 평가 F1 → MAE. 결과 MAE 0.272(5점 만점에 0.27점 오차). 의미: "코가 크다"가 아니라 "친근함 4.2점, 카페 적합도 87점". 비즈니스 가치가 즉시 명확해졌다.

[10개 인상 라벨 매칭 업종] 친근한→카페·식당, 신뢰감→병원·금융·법률, 전문적인→컨설팅·IT·교육, 따뜻한→육아·펫·요양, 세련된→뷰티·패션·인테리어, 활기찬→헬스·스포츠·배달, 차분한→서점·명상·차, 귀여운→디저트·캐릭터·키즈, 강인한→건설·이사·자동차, 지적인→학원·출판·테크.

[규칙 기반 라벨 변환] 20만 장을 LLM으로 다 라벨링하면 비용이 크니까, 물리 속성 12개를 가중치 규칙으로 조합해 인상 점수 10개로 즉시 변환했다. 예: Smiling + Big_Lips + High_Cheekbones → 친근함 ↑. 그 후 Gemini로 1,000장만 검증해 룰을 보정하는 전략.

[ImpressionMLP v4 — 연예인 데이터 보정] 가장 임팩트 큰 개선. 핵심 아이디어: 대기업이 유재석을 은행 광고에 쓴다 = "금융에는 유재석 같은 인상이 효과적"이라고 수억 써서 이미 검증한 것. 그걸 무료로 가져다 쓰자. 15개 업종 50명+ 연예인 광고 출연 이력 수집 → GPT-4o-mini로 인상 평가($0.15) → Ridge Regression으로 매핑 가중치 자동 학습 → CelebA 20만 장 재라벨링 → 모델 재학습. 결과 MAE 0.272 → ~0.002.

[Ridge Regression 매핑 보정] "감으로 만든 가중치"를 버리고 "데이터 학습 가중치"로 교체. Before: "Smiling 가중치 0.7이면 되겠지." After: Ridge가 GPT 평가 데이터에서 통계적으로 최적 가중치 산출. 이게 머신러닝 시스템에서 흔히 빠지는 함정 — 사람이 임의로 정한 상수가 실은 학습되어야 할 파라미터인 경우.

[MAE 0.002의 의미] 매우 낮은 오차지만 함정도 있다. 보정된 Ridge 규칙이 물리 속성→인상 점수를 매우 일관되게 변환하므로, 모델이 "랜드마크 → 물리 속성 추론 → 인상 점수"를 사실상 결정론적으로 학습한 것. 새 얼굴이 들어와도 일관된 점수가 나오긴 하지만, 라벨 자체가 규칙 기반이라 "진짜 인상"보다 "규칙이 정의한 인상"을 잘 맞추는 셈.

[연예인 인상 평가 비용] 연예인 50명 × 10라벨을 GPT-4o-mini로 평가하는데 약 $0.15(200원). 작은 비용으로 검증된 라벨을 확보. 후에 v8 vision 라벨링은 367명까지 확장.

[Muse v2 FLUX LoRA] 베이스 FLUX.1-dev에 한국인 얼굴 특성을 주입하는 LoRA 어댑터. 파일 크기 344MB, Rank 32, Alpha 32, 1500 스텝, 3시간, A100 40GB. 데이터: AIHub 71415(안면 인식 에이징) 200명분 18GB 다운로드 → 사람당 5장 × 200명 = 1,000장 큐레이션 → 512px 이하 드랍 → 1024×1024 리사이즈 → 최종 877장 학습.

[FLUX LoRA 학습 환경] Colab Pro+ A100 40GB. 프레임워크 ai-toolkit (ostris). 옵티마이저 AdamW 8-bit, LR 1e-4, dtype bf16, 샘플러 flowmatch. 총 35 compute units 소모. EMA decay 0.99.

[한국인 LoRA 효과] Without Muse LoRA: 순수 FLUX는 서양인 편향 강함, 한국인 프롬프트 줘도 어색. With Muse LoRA(scale 0.5): 한국인 골격/피부톤/눈매 자연스럽게 반영. Realism LoRA(0.8) + Koda LoRA(0.3)와 함께 쓸 때 최적. CFG 3.0 / 40~45 step에서 최고 품질.

[AIHub 71415 데이터 확보] 한국인 얼굴 데이터셋은 라이선스 제약이 크다. AIHub 71415는 내국인 신청·승인제. 신청서 제출 → 자동 승인 → aihubshell CLI로 18GB 다운로드 → ZIP 분석 → 사람당 5장 큐레이션. 처음에는 더 큰 K-Face 데이터셋을 노렸지만 용량 문제로 71415로 우회.

[학습 데이터 큐레이션] AIHub 71415 200명 × 5장 = 1,000장 원본에서 512px 이하 드랍, 센터 크롭 후 1024×1024 리사이즈. 최종 학습 세트 877장. 데이터 전처리 → tar.gz 압축 → HuggingFace private dataset repo 업로드 → Colab에서 HF pull → 학습 → LoRA를 HF 공식 repo에 push. 1회성 파이프라인이지만 재현 가능하게 짜둠.

[Muse 광고 생성 파이프라인 4-Pass] Pass 0: 카탈로그 DB 조회(모델 인상 프로필, 추천 업종 top 3, 사전 생성 베이스 이미지). Pass 1: FLUX + Muse v2 LoRA + Realism + Koda → 자연스러운 업종 씬. Pass 2: 제품 이미지 있으면 img2img/inpainting으로 합성. Pass 3: GPT-image-1.5로 광고 디자인(로고/카피 배치). Pass 4: PIL 후처리(로고 정확도 보정, 텍스트 오타 교정). 최종 9:16 광고 이미지 출력.

[광고 합성 5가지 일관성 차원] "디자인 같다 vs 붙여놓은 것 같다"의 차이는 5가지 물리 일관성에서 나온다: (1) 원근/소실점 정렬, (2) 조명 방향과 질, (3) 스케일, (4) 피사계심도, (5) 색온도. 현재 diffusion 도구는 3-4개를 자동 처리하지만, 정확한 specular highlight 매칭과 sub-pixel shadow geometry는 여전히 자동 합성이 약한 영역.

[IC-Light 활용] Lvmin Zhang의 IC-Light(2024)는 product cutout을 reference scene 조명에 맞춰 relighting 해준다. Pass 2 단계에서 제품을 씬에 자연스럽게 녹이는 데 사용. 2025 시점에서 실용 수준에 도달한 도구.

[단위 경제성] 광고 이미지 1장당 비용. Pass 1(FLUX + LoRA): $0.05(fal.ai) 또는 $0.02(자체 GPU). Pass 2(제품 합성): $0.05 또는 $0.02. Pass 3(GPT-image): $0.04~0.08. 합계 $0.14~0.22/장. 월 10,000장 기준 140만~220만원. 자체 GPU 운영 vs API 호출의 손익분기점은 월 5,000장 근처.

[fal.ai 활용] FLUX + LoRA 서빙 API. flux-lora 엔드포인트는 $0.05/이미지. face-swap(InsightFace 기반) $0.03/이미지. 자체 GPU 인프라 없이 빠르게 프로토타입을 돌려보고 트래픽 보고서 자체 인프라로 전환하는 게 초기 스타트업 정석.

[KOBACO 광고 데이터 리서치] KOBACO AiSAC API에 36,000+ 광고 DB(업종/인물/감정/키워드)가 있다. 한 번에 50개씩만 반환되는 제약 때문에 대규모 수집은 어려움. 공공데이터포털 CSV는 폐기 상태. 결국 수동 + 학술 메타분석으로 보강.

[광고효과 학술 메타분석] 한양대 175편 메타분석: 모델-제품 이미지 일치 시 광고효과 24.8% 증가. Springer 46편: endorser credibility × product type 효과. 전문가 모델은 비전문가 대비 9.5% 효과 증가(고관여 제품에서). 유명인 모델은 저관여 제품에서 더 효과적. 매력성은 매력 관련 제품(패션, 화장품)에서만 효과, 무관 제품에서는 효과 없음.

[관여도 × 소구유형 매트릭스] 고관여+이성(금융/의료/법률/건설) → 전문성·신뢰성. 고관여+감성(패션/화장품/예술/IT) → 세련됨·매력성. 저관여+이성(교육/생활서비스/농업) → 친근함·신뢰성. 저관여+감성(식당/카페/주류/스포츠) → 친근함·활발함. NCS 23개 + 소상공인 14개 = 37개 업종 매핑.

[Colab GPU 시간 제약] FLUX 모델 다운로드(33GB)를 여러 번 반복하면 GPU 시간이 빠르게 소진된다. 학습 데이터셋이 너무 크면 latent 캐싱 단계에서 OOM/타임아웃. 해결: 데이터를 3,000장 이하로 축소, FLUX는 Drive 마운트로 1회 다운로드 후 재사용, 학습 노트북을 작은 셀로 쪼개서 중단 재개 가능하게.

[중단 재개 패턴] CelebA 20만 장 처리에 3-5시간이 걸리는데 Colab 세션이 끊기면 처음부터 다시 해야 했다. 해결: 처리 완료된 파일명을 SQLite DB에 기록 → 재시작 시 DB에서 처리 안 된 것만 골라서 진행. ML 파이프라인의 모든 장시간 작업은 idempotent해야 한다.

[Google Drive를 영속 레이어로] Colab은 휘발성. SQLite DB 파일을 Google Drive에 두고 마운트해서 쓰면 세션 끊겨도 데이터 유지. rclone이 권한 문제로 차단되면 Python Google Drive API로 우회. 작은 인프라일수록 의존성 줄이고 표준 도구로 가기.

[HuggingFace Hub 활용] 학습된 모델 → musecall/face-impression-analyzer로 push. 추론 시 ImpressionPredictor가 HF에서 모델 다운로드 → MediaPipe로 랜드마크 추출 → 추론. 모델/데이터셋을 private repo로 두면 팀 협업이 쉬워진다.

[CelebA의 한계] 20만 장으로 충분한 듯 보이지만 서양인 편향. 한국 시장 광고에는 부적합. 한국인 얼굴 학습을 위해 AIHub 71415가 필요했고, 그것도 200명 18GB로 압축해 877장만 사용. 데이터 양보다 품질·도메인 적합성이 더 중요하다.

[CNN 전환 검토] MLP + 랜드마크 28개의 기하 정보로 F1 0.58이 한계. 피부 질감/색상/머리카락 같은 시각 정보를 못 쓴다. 옵션: (1) 현재 MLP로 출시, (2) MobileNet CNN 전환(F1 0.75+, GPU 더 필요), (3) MLP + CNN 앙상블(F1 0.70+). 초기 스타트업이라면 (1) → 트래픽 보고 (2)/(3) 결정이 합리적.

[MediaPipe 468 랜드마크] Google 라이브러리. 얼굴에서 468개 점을 빠르게 추출. CPU에서도 실시간 동작. v3 이후 우리는 28개 핵심점만 사용(눈/코/입/윤곽). 랜드마크 정밀도는 충분하지만, 표정 변화나 측면 얼굴에서 흔들림이 생긴다.

[Focal Loss 도입] v3에서 추가. 쉬운 샘플은 무시하고 어려운 샘플에 집중하는 손실 함수. 클래스 불균형 데이터에서 일반 BCE보다 잘 동작. 단점: 하이퍼파라미터(γ) 튜닝이 필요하고, 너무 크게 잡으면 학습이 불안정.

[속성별 threshold 탐색] 모든 속성에 0.5를 쓰는 건 차선. 각 속성마다 최적 threshold를 validation에서 찾아야 한다. 우리는 0.05 단위로 0.05~0.95 범위 그리드 서치. F1 기준 최적값 사용. 추론 시 thresholds.json을 같이 배포해 정확히 같은 분기점 적용.

[데이터 augmentation — 랜드마크 노이즈] 이미지 augmentation 대신 랜드마크 좌표에 가우시안 노이즈 추가. 표준편차 0.005~0.01 정도. 효과: 모델이 미세한 좌표 변동에 강건해짐. 이미지 변환보다 빠르고, 캐싱된 랜드마크에 즉시 적용 가능.

[연예인 광고 모델 데이터 수집] 15업종 × 50명+ 연예인이 어떤 브랜드 광고에 출연했는지 웹 검색으로 조사. 예: 은행은 박보검(신한), 아이유(우리), 장원영(우리), 임영웅(하나). 주류는 아이유(참이슬), 공유(테라), 카리나(크러시). 치킨은 변우석(교촌), 변요한(BBQ), 르세라핌(굽네). 공개 광고 캐스팅 정보라 초상권 침해 없음.

[v8 Vision 라벨링 확장] GPT-4o Vision으로 367명 × 10라벨 재평가. v3의 GPT-4o-mini text 평가에서 한 단계 발전. Vision 입력은 비용이 높지만 라벨 품질이 안정적. 카탈로그 선정 풀로 활용.

[카탈로그 60명 구조] 사전 생성 모델 카탈로그. 5M(남) + 5F(여) × 6라벨 = 60명. 라벨: friendly/trustworthy/professional/sophisticated/warm/energetic. 임의 한국 이름 자동 생성. 모델당 시드 고정으로 일관성 유지. 6개 노트북을 라벨별로 병렬 실행.

[프롬프트 설계 패턴] 인물 이미지 생성 프롬프트는 "editorial magazine portrait of [subject], celebrity-level attractiveness, perfect skin, [top2_impression_keywords], [pose_description], kodak portra 400, 35mm, film grain, cinematic tones" 구조. 성별별 subject만 분기. 카메라/필름 키워드가 톤 결정에 가장 중요.

[광고 레퍼런스 필터링] 1차 875장 광고 이미지 수집 → 1차 통과 162장 → 2차 엄격 필터로 162장만 A급 채택. 모델 입장에서 학습이 아니라 GPT Pass 3에 참조용으로만 사용 → 저작권 fair use 범위.

[Phase 2 일반인 풀 구축 계획] 본인 얼굴 업로드 → ImpressionMLP가 자동 인상 분석 → 업종 자동 추천 → 매칭. 일반인 모델 로열티 표준안 마련 예정. 데이터가 쌓이면 추천 정확도 향상이 진짜 해자.

[성과 데이터 기반 매핑 자동 보정] 광고 조회수/클릭률 데이터를 매핑 가중치 학습에 다시 투입하는 폐쇄 루프. 처음에는 GPT 평가 → Ridge로 시작했지만, 진짜 검증 데이터는 시장에서 나온다. 운영 시간이 곧 모델 정확도가 되는 구조 설계.

[풀스택 → AI 전환의 어려움] 프론트(Next.js)와 백엔드(Node.js)는 동기/요청-응답 패턴이지만, ML은 학습-추론 분리 + 모델 버전 관리 + 배치 vs 실시간 분기까지 필요. Python 생태계 학습 + 클라우드 GPU 비용 감각 + ML 라이브러리(PyTorch, Diffusers, MediaPipe) 습득이 동시에 와서 첫 3개월이 가장 힘들었다.

[LLM을 서비스에 붙인 경험] 풀스택 시기에 자사 서비스에 LLM 연결. 문제: 응답 시간 변동성, 비용 예측 불가, 출력 형식 보장. 해결: streaming UI로 체감 응답속도 개선, prompt caching으로 비용 절감, JSON 모드 또는 function calling으로 형식 강제. 중요한 건 LLM이 종종 틀린다는 가정 하의 검증 레이어.

[프롬프트 엔지니어링 공식] [역할] + [맥락] + [작업] + [제약조건] + [형식]. 예: "당신은 SEO 전문가다(역할). 이 블로그는 AI 도구를 다룬다(맥락). 롱테일 키워드 10개를 추천하라(작업). 검색량 100-1,000 범위(제약). 표 형식으로(형식)." 출력 일관성이 극적으로 올라간다.

[1인 운영 워크플로 — RAG 자동 발행] Vercel cron + Gemini + pgvector. 매일 새벽 AI가 글 5-10개 생성 → DRAFT 저장 → 예약 시간에 자동 PUBLISHED → IndexNow 자동 제출 → 검색엔진 즉시 인덱싱. 사람 손이 닿는 건 카테고리 분류와 최종 검토뿐.

[자동 발행의 함정] 매일 자동 발행하면 발행 패턴이 부자연스러워 SEO에 불리. 해결: 생성 vs 발행 분리, 발행은 간격 두기(주 2-3회), 사람 검토 게이트 필수. AI 생성 페널티를 피하려면 "확장된 콘텐츠 악용" 정책 위반 안 되게 독창성·가치 추가 필수.

[비용 관리 — Gemini Flash vs Pro] 콘텐츠 생성에는 Gemini 2.5 Flash Lite로 충분. Pro는 복잡한 코드/리서치만. 일일 5-10개 글 생성 비용은 Flash Lite 기준 월 $5 미만. 임베딩은 text-embedding-004로 별도 처리.

[초기 스타트업 멀티롤] 모델 학습 + API 서빙 + 프론트엔드 + 데이터 큐레이션 + 마케팅까지 한 사람이 전부. 우선순위 결정이 가장 중요. 황금 규칙: "단위 경제가 안 맞으면 다 멈추고 그것부터." 예쁜 모델보다 장당 $0.20 이하가 먼저.

[학습 가속법 — Feynman + Active Recall] 새 ML 개념을 익힐 때: (1) 누군가에게 설명하듯 글로 정리(Feynman), (2) Anki로 spaced repetition, (3) 책 닫고 기억 테스트(active recall). FLUX, LoRA, MediaPipe를 3개월 만에 운영 수준으로 익힌 방법.

[Reading Strategy] 논문은 abstract → figures → conclusion → method 순서로 읽으면 시간 절약. 코드가 있으면 코드부터 읽고 논문은 confirmation으로. 재현 안 되는 논문은 시간 낭비라 드롭.

[디자인 결정 기록] 모든 비자명한 결정은 docs/devlog.md에 날짜와 함께 기록. "왜 듀얼헤드를 택했나, 왜 45특징인가, 왜 v3에서 회귀로 바꿨나." 6개월 후의 나는 그때의 나를 기억하지 못한다.

[테스트 전략 — AI API의 비결정성] Gemini는 매번 다른 결과를 반환. 해결: (1) 형태 검증(JSON 구조, 필드 타입, 길이), (2) Fuzzy matching(정확 텍스트 X, 패턴 매칭), (3) MSW로 mocking, (4) Golden outputs(실제 응답 스냅샷 → 회귀 테스트). 정확 일치 검증은 무조건 깨진다.

[버전 관리 — 모델/데이터/코드 3축] 모델은 HuggingFace tag, 데이터는 dataset repo + 해시, 코드는 Git. 셋이 동기화되어야 재현 가능. v1/v2/v3/v4 모델 경로가 다르고 라벨 정의가 다르면 추론 코드도 분기. 추론 코드는 항상 모델 버전을 체크하도록.

[N잡 자동화 — 블로그 + YouTube] Vercel 블로그 자동 생성 + 인기 글 → Shorts 스크립트 자동화 + 한↔영 자동 번역. 한 번에 다 만들지 말고 한 파이프라인 고도화 후 다음. 처음 3개월은 발행만, 다음 3개월은 분석, 그 다음에 수익화.

[1인 SaaS 시작점] 본인이 매주 시간을 쓰고 있는 작업이 정답. 추측으로 만들지 말고 본인 워크플로 자동화부터. 블로그 자동화도 그렇게 시작.

[AI 도구 평가 기준] (1) 워크플로에 자연스럽게 들어오나, (2) 잠깐 빛나고 끝나나 1개월 후도 쓰나, (3) 비용 대비 시간 절감 vs 학습 곡선. "신기해서" 쓰는 도구는 한 달 안에 사라진다.

[솔로 개발자 번아웃 방지] (1) 작업 시간 블록(90분 + 휴식), (2) 한 번에 한 모델만, (3) Friday는 코드 안 만지고 회고/문서화, (4) 일요일은 완전 차단. 학습 곡선이 가파른 1년차일수록 회복 시간이 더 필요.

[기록의 중요성] 이 블로그가 곧 나의 외부 메모리. devlog 패턴 그대로 글로 옮기면 자동으로 1차 자료가 된다. 잘못된 결정도 적어둔다 — 미래의 나에게 가장 중요한 정보다.
